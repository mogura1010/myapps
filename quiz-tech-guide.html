1全体像 — データと画面の関係

quiz.html は「データ（問題バンク）」と「画面（3つのカード）」を JavaScript が橋渡しする構造になっています。

画面は3つのカード（同時に1つだけ表示） ┌─────────────────────────────────┐ │ #start（ジャンル選択） │ ← 最初に表示 │ #quiz（出題中） │ ← 選択後に表示 │ #result（結果表示） │ ← 全問終了後 └─────────────────────────────────┘ ↕ JavaScriptが切り替え ┌─────────────────────────────────┐ │ BANK オブジェクト（問題データ） │ │ vbm: { questions: [...] } │ │ tmvpn: { questions: [...] } │ │ tmc: { questions: [...] } │ └─────────────────────────────────┘

2データ設計 — BANKオブジェクトの構造

すべての問題データは BANK という1つのオブジェクトにまとめられています。設計の工夫を読み解きましょう。

BANKの構造

const BANK = {
  vbm: {
    name: "VBM / iVBM（ウイルスバスター モバイル）",
    questions: [
      {
        q: "トライアルライセンスの有効期間は？",
        choices: ["7日間", "14日間", "30日間", "90日間"],
        answer: 2,     // ← 正解は choices[2]（30日間）
        explain: "体験版は30日間のみ有効..."
      },
      // ... 他の問題
    ]
  },
  tmvpn: { /* 同じ構造 */ },
  tmc:   { /* 同じ構造 */ }
};

answer は「インデックス番号」で持つ

answer: 2 は「choices配列の2番目（0始まり）が正解」という意味です。正解の文字列そのものではなく 番号で持つ のがポイントで、後でシャッフルするときに対応が取れます（セクション3で詳しく説明）。

設計の選択肢メリットデメリット
正解を文字列で持つ（"30日間"）直感的に読めるシャッフル後も文字列比較が必要、タイポで壊れる
正解をインデックスで持つ（2）← 採用シャッフル後の再マッピングが可能コード読解に慣れが必要

Object.keys() でジャンル一覧を動的生成

Object.keys(BANK).forEach(key => {

3シャッフルアルゴリズム — Fisher-Yates法

問題の順番と選択肢の順番を毎回ランダムにするために、Fisher-Yates シャッフル を使っています。

Fisher-Yatesとは？

Math.random() で単純にソートする方法（arr.sort(() => Math.random() - 0.5)）は統計的に偏りが出ます。Fisher-Yates は末尾から順に「ランダムな位置と交換」を繰り返す方法で、すべての並び順が等確率になります。

function shuffle(arr) {
  const a = arr.slice();  // 元の配列を壊さないようコピー
  for (let i = a.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [a[i], a[j]] = [a[j], a[i]];  // 分割代入で交換
  }
  return a;
}
例：[A, B, C, D] をシャッフル i=3: j=1 → [A, D, C, B] ← B と D を交換 i=2: j=2 → [A, D, C, B] ← C と C を交換（変化なし） i=1: j=0 → [D, A, C, B] ← A と D を交換 結果: [D, A, C, B]

選択肢シャッフル後のanswerインデックス再マッピング

選択肢をシャッフルすると「正解が何番目か」が変わります。shuffleChoices() 関数がこの再マッピングを担当しています。

function shuffleChoices(item) {
  // [0,1,2,3] をシャッフルした順序を作る
  const order = shuffle(item.choices.map((_, i) => i));
  // 例: order = [2, 0, 3, 1]

  return {
    q: item.q,
    choices: order.map(i => item.choices[i]),
    // order.indexOf(item.answer) で「元の正解が新しい配列の何番目か」を求める
    answer: order.indexOf(item.answer),
    explain: item.explain
  };
}
具体例： 元の choices = ["7日間", "14日間", "30日間", "90日間"] 元の answer = 2（"30日間"） order = [2, 0, 3, 1] にシャッフルされた場合 新しい choices = ["30日間", "7日間", "90日間", "14日間"] 新しい answer = order.indexOf(2) = 0（先頭が正解）

答え後の色変え

// choose(i) 内 — 全ボタンを走査して色を付ける
buttons.forEach((b, bi) => {
  b.disabled = true;  // 二重選択を防ぐ
  if (bi === item.answer) {
    b.style.background = "var(--ok-bg)";  // 正解は緑
  } else if (bi === i) {
    b.style.background = "var(--ng-bg)";  // 選んだ不正解は赤

locked 変数の役割

答えを選んだ瞬間に locked = true にして、2度目のクリックを if(locked) return; で弾いています。disabled を付けるより先にロックすることで、高速連打でも二重処理されません。

5CSS — ダークモード・プログレスバー・safe-area

quiz.html は見た目の工夫が3つあります。CSS変数によるダークモード、アニメーション付きプログレスバー、iPhoneのノッチ対応です。

CSS変数でダークモードを一括切り替え

/* ライトモード（デフォルト） */
:root {
  --bg: #f4f3ef;
  --accent: #d8392f;  /* トレンドマイクロの赤 */
}

/* ダークモード — OSの設定を自動検知 */
@media (prefers-color-scheme: dark) {
  :root {
    --bg: #1c1c1a;
    --accent: #f06a60;  /* 暗い背景でも見やすい明るめの赤 */
  }
}

var(--accent) と書いておくだけで、ダークモード時は自動的に明るい赤に切り替わります。ボタン・ボーダー・テキストなど何十箇所あっても、変更は :root の2行だけです。

プログレスバーのアニメーション

/* CSS側 — widthの変化を0.3秒でアニメーション */
.track > div {
  height: 100%; width: 0%;
  background: var(--accent);
  transition: width .3s;
}

// JS側 — 幅をパーセントで指定するだけ

safe-area — iPhoneのノッチ・ホームバー対応

/* iPhoneのノッチ・Dynamic Islandにコンテンツが隠れないよう余白を確保 */
body {
  padding: env(safe-area-inset-top)
           env(safe-area-inset-right)
           env(safe-area-inset-bottom)

}
iPhone の画面 ┌──────────────────┐ │ ▓▓▓ Dynamic Island ← safe-area-inset-top で避ける │──────────────────│ │ │ │ コンテンツ │ │ │ │──────────────────│ │ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓ ← safe-area-inset-bottom で避ける └──────────────────┘ （ホームバー）

viewport-fit=cover をmetaタグに付けると画面端まで使えますが、その代わりにsafe-areaの指定が必要になります。quiz.htmlはこのセットで正しく実装されています。

次に学ぶ技術：fetch API、async/await、CORS

よく使われるダメなシャッフル

// NG: sort + random は統計的に偏りがある
arr.sort(() => Math.random() - 0.5);

このコードは「比較関数が毎回ランダムな値を返す」ことで並び順を変えますが、sort の内部アルゴリズムによっては特定の順番が出やすくなります。大事な場面（テスト・ガチャなど）では使わないほうが無難です。

Fisher-Yatesが正しい理由

// 末尾から先頭に向かって「ランダムな位置と交換」
for (let i = a.length - 1; i > 0; i--) {
  const j = Math.floor(Math.random() * (i + 1));
  [a[i], a[j]] = [a[j], a[i]];  // 分割代入で交換
}

各ステップで「残りの要素から等確率に1つ選ぶ」を繰り返すので、すべての並び順が数学的に等確率になります。

📚 ステップアップ

Fisher-Yatesで使っている [a[i], a[j]] = [a[j], a[i]] は 分割代入 という構文です。

// 従来の交換（temp変数が必要）
let temp = a[i];
a[i] = a[j];
a[j] = temp;

// 分割代入での交換（1行で書ける）
[a[i], a[j]] = [a[j], a[i]];

分割代入はオブジェクトにも使えます：

const { name, questions } = BANK.vbm;
// name = "VBM / iVBM..."
// questions = [...]

ブラウザのコンソール（F12）で以下を実行すると、各並び順の出現回数を検証できます。

// Fisher-Yates の検証スクリプト
const counts = {};
for (let t = 0; t < 10000; t++) {
  const a = [0, 1, 2];
  for (let i = a.length - 1; i > 0; i--) {
    const j = Math.floor(Math.random() * (i + 1));
    [a[i], a[j]] = [a[j], a[i]];
  }
  const key = a.join(",");
  counts[key] = (counts[key] || 0) + 1;
}
console.log(counts);
// 各パターンが約1666回（=10000÷6）になれば均等

クロージャとは何か

JavaScriptの関数は「作られたときのスコープを記憶する」性質があります。これをクロージャと呼びます。

// forEach のループ変数 i が各ボタンに閉じ込められる
item.choices.forEach((c, i) => {
  const b = document.createElement("button");
  b.textContent = c;
  b.onclick = () => choose(i);  // この i は 0, 1, 2, 3 それぞれ固定
  box.appendChild(b);
});
⚠️ よくあるミス：var でループするとクロージャが壊れる
// NG: var は関数スコープなので全ボタンが同じ i を共有してしまう
for (var i = 0; i < choices.length; i++) {
  buttons[i].onclick = () => choose(i);  // 全部 i=4 になる！
}

// OK: let はブロックスコープで各反復に独立したiを作る
for (let i = 0; i < choices.length; i++) {
  buttons[i].onclick = () => choose(i);  // 正しく 0,1,2,3 になる
}

querySelectorAll と forEach

// 選択後に全ボタンを操作する場面

📚 ステップアップ

quiz.htmlは各ボタンに個別に onclick を付けています。ボタンが増えるほどイベントハンドラも増えますが、イベント委譲 を使うと親要素1つにまとめられます。

// 各ボタンにonclickを付ける代わりに、親要素で受け取る
$("choices").addEventListener("click", (e) => {
  const btn = e.target.closest("button");
  if (!btn) return;  // ボタン以外は無視
  const i = [...btn.parentNode.children].indexOf(btn);
  choose(i);
});

動的にボタンを追加・削除してもイベントが自動的に効くのがメリットです。

⚠️ よくあるミス1：getElementById で存在しない要素を取得してnullエラー

// デバッグ：まずconsole.logで確認
console.log($("progres"));  // → null と表示されればIDが間違い
⚠️ よくあるミス2：イベントを二重登録してしまう
// render()を呼ぶたびにaddEventListenerが追加される
$("next").addEventListener("click", handler);  // render()の中に置くとNG

// quiz.htmlは $("next").onclick = ... と代入方式にして回避している
$("next").onclick = () => { ... };  // 代入は上書きされるので二重登録なし

DevToolsでの確認手順：

  1. F12 → Elements タブ → 対象の要素を選択
  2. 右ペインの「Event Listeners」タブで登録済みイベントを確認
  3. 二重登録されていれば同じ関数が2行表示される
